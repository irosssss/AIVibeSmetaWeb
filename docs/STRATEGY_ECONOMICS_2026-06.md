## 0. Главный технический вердикт (определяет движок и себестоимость)

**YandexART умеет только text→image — он не редактирует загруженное фото** (подтверждено по
официальному `yandex-ai-studio-sdk`, обновлён 23.06.2026: на вход только текст, ни `image`, ни `mask`).
**Kandinsky 3.1 / GigaChat** модель inpainting умеет, **но публичный API в РФ его не отдаёт** (только
text2image). Управляемого img2img/inpaint в managed-API РФ на 2026 фактически нет.

**Решение:** ядро стейджинга = **self-hosted SDXL 1.0 inpainting + multi-ControlNet** (либо
отечественный **Kandinsky 3.1**, Apache-2) на **Yandex Compute GPU**. Это RU-резидентно (критично под
152-ФЗ для риелторов/застройщиков) и это главная — и единственная незаменимая — статья себестоимости.

### 1.2. Себестоимость продукта (COGS)

- **1 стейджинг = ~4 кадра** (4 стиль-варианта на согласование).
- Чистое GPU-время: **~7–11 ₽/кадр на A100**, **~0,5–2 ₽/кадр на T4**.
- С учётом простоя пула + апскейл (Real-ESRGAN): **~10–40 ₽ за стейджинг**.
- Токены YandexGPT на стейджинг — **< 0,5 ₽** (пренебрежимо).
- **Маржа стейджинга ~90–95%.** При объёме 10 000 стейджингов/мес: ~400к ₽ GPU на on-demand A100
  или **~100–150к ₽** на прерываемых/T4-пуле с батчингом.
- ⚠️ Главный скрытый расход — **GPU в простое**, не сам инференс. Лечится прерываемыми ВМ + очередью +
  autoscale-to-zero + предочисткой LaMa на CPU (не каждый шаг требует GPU).

### 1.6. Три рычага окупаемости

1. **Рамп и цена подписки риелторов** — главный (LTV/CAC 5–20×, маржа ~95%): +10% к скорости набора
   или +1000 ₽ к чеку сдвигает безубыточность на 1–2 мес раньше и срезает яму на 1–2 млн ₽.
2. **Режим GPU** — переход с on-demand A100 на прерываемые ВМ + батчинг + autoscale-to-zero роняет
   COGS с ~18 до ~6–10 ₽/кадр → сдвиг кумулятивной окупаемости с ~М30 к М23.
3. **Тайминг найма** — держать команду 6 человек, дизайн/поддержку на подряде, нанимать только под
   подтверждённую выручку: −1,5–2 млн ₽ к пиковой яме. Экономить нельзя только на ML-инженере и
   GPU-сервисе (ядро тарана).

> CAC (оценки, прямых RU-бенчмарков нет): риелтор — CPL 1500–4000 ₽, CAC платящего ~5–20к ₽,
> LTV/CAC 5–20×; дизайнер — CAC 8–30к ₽, LTV/CAC 1–4× (рентабелен только при годовом биллинге).
> Масса не привлекается платно (LTV/CAC 0,03–0,38, убыток).

## 2. SaaS для дизайнеров — детальный спек

**Ядро ценности:** «От фото комнаты до утверждённого клиентом дизайна со спецификацией и сметой — за
один вечер, а не за две недели». Закрываем две самые дорогие боли: согласование концепции (3–4 фото
варианта стиля за минуты вместо часов в 3ds Max) и спецификацию/смету (40–100 ч/проект вручную →
авто-смета в ₽ с живыми ценами фабрик). **Уникум, которого нет ни у Foyr, ни у Spacely, ни у
Planoplan:** единая цепочка `рендер → расстановка по нормам → спецификация → смета → закупка` без
ручного дублирования в Excel + RU-резидентность + оплата картами РФ. Первый «вау» < 10 минут.

### 2.5. UX-каркас веб-кабинета (рабочие правила)

> Источник: разбор статей [uprock.ru](https://www.uprock.ru/all-articles) (2026-06-25). Это не фичи, а правила
> поведения кабинета поверх модулей 0–6 — они определяют, поверит ли дизайнер ИИ-результату настолько, чтобы
> показать его клиенту. Пользовательские/мобильные эквиваленты (онбординг, ошибки, уведомления, степпер
> ожидания) — в [`UX_GROWTH.md`](UX_GROWTH.md) §9. Усилие: **S** = часы–день · **M** = дни · **L** = неделя+.

- [ ] **(M) «Проверь и подтверди» перед финализацией.** Никакого автокоммита ИИ-вывода в смету / спецификацию / листинг: diff-предпросмотр («ИИ предложил эти 8 позиций, эту планировку, эту сумму») с правкой каждого элемента. Модули **4–5**. *Слепой copy-paste вывода ИИ = риск для бизнеса. («Умное тестирование»)*
- [ ] **(M) Журнал дизайн-решений.** Каждый ИИ-концепт (стиль, расстановка, замена позиции) несёт краткую аргументацию «проблема → что предложено → почему (эргономика / бюджет / стиль)». Модули **1–2**. Лечит «чёрный ящик» — дизайнер защищает решение перед клиентом своими словами. *(«Как принимать дизайн-решения»)*
- [ ] **(S) Матрица выбора вариантов.** При 3–6 равноценных концептах — мини-сравнение по осям (соответствие брифу · бюджет · эргономика · «вау»), а не голая галерея. Снимает паралич выбора. Модуль **1**. *(там же)*
- [ ] **(M) Undo-first вместо модалок.** Обратимые действия (удалить проект / мудборд / позицию сметы) — тост «Отменить» + soft-delete на бэке. Полноэкранное подтверждение оставить **только** для необратимого и денежного: публикация листинга, отправка заказа фабрике (CPA) — там же дисклеймер «AI-стейджинг». Модули **5–6**. *(«Окна подтверждения»)*
- [ ] **(S) «Избранное» ≠ «в проект».** Две разные механики, не одна звёздочка: «Моя библиотека» — личный вкус дизайнера между проектами (без таймера); «В проект / на мудборд» — рабочий шаг. Модуль **1**, каталог. *Кейс-ошибка Slack: одна иконка на два смысла → костыли. («Закрепление ≠ сохранение»)*
- [ ] **(M) Замыкание циклов в «Сохранённом».** ИИ-черновики и отложенные концепты — со статусами (Новое → В работе → Применено / Отклонено) и счётчиком висящих; список умеет **закрывать** циклы, а не только копить (эффект Зейгарник). Дашборд. *(там же)*
- [ ] **(S) Pin для командных пространств.** Утверждённый рендер / бриф по объекту закреплён наверху для всей команды (риелтор + застройщик + фотограф) — видимость без поиска. Командная механика, не личное избранное. Модуль **3** / общий проект. *(там же)*
- [ ] **(S) Индикатор прохождения цепочки.** Видимый прогресс проекта бриф → концепт → расстановка → смета → передача (даёт чувство компетентности и автономии). Дашборд. *(«Эмоциональные потребности»)*
- [ ] **(L) Единый визуальный язык.** Одна токенизированная type-scale + гештальт-группировка карточек (тень = слой, близость = группа, линия = этап цепочки) — против рассинхрона при двух платформах и трёх ICP. Это профилактика UX-долга (см. `UX_GROWTH.md` §5). *(«Типографическая иерархия», «Гештальт»)*

**Приоритет (эффект/усилие):** сначала «Проверь и подтверди» (4–5) и Undo-first (5–6) — они прямо защищают деньги дизайнера перед клиентом и фабрикой; затем журнал решений и замыкание циклов; единый визуальный язык — когда сядем за дизайн-систему.

## 3. Виртуальный стейджинг — технический пайплайн

### 3.1. Как определяется стиль (ключевой вопрос)

Стиль **не определяет отдельная нейросеть-классификатор.** Стиль **задаёт пользователь**, а «переодевает»
комнату генеративная модель (SDXL). Три механизма:

1. **Пресет + текст-промпт** (MVP) — выбрал «скандинавский / современная классика / лофт / минимализм /
   неоклассика / светлый под аренду» → разворачивается шаблонный 4-частный промпт. Дёшево, предсказуемо,
   повторяемо.
2. **Референс-фото через IP-Adapter** (премиум, фаза 2) — даёшь образец/мудборд → «сделай как на этой
   картинке» (точное попадание в бренд риелтора или мудборд дизайнера).
3. **Style-LoRA** — обученный адаптер под фирменный пресет (стабильное узнаваемое «лицо» материалов/света).

YandexGPT 5 играет вспомогательную роль — классифицирует стиль по тексту и собирает качественный промпт,
**но саму картинку не редактирует.** Под два ICP: риелтор → пресеты + LoRA (предсказуемость, скорость);
дизайнер → + IP-Adapter (свой мудборд).

### 3.2. Пайплайн (7 шагов)

| # | Шаг | Модель | Что делает |
|---|---|---|---|
| 1 | Приём, валидация | backend + лёгкий CV | резкость, NSFW, нормализация; задача в очередь к GPU-сервису |
| 2 | Геометрия: глубина + линии | **Depth Anything V2** + MLSD/Canny | depth-карта (перспектива) + рёбра стен/окон → conditioning |
| 3 | Маски пол/стены/окна | **Grounding DINO + SAM 2** | inpaint-маска = пол; protected-маски = окна/стены (туда не пишем) |
| 4 | **План расстановки** | **AIVibe ArrangementEngine** (геометрия, не ИИ) | координаты/габариты в метрах → проекция в перспективу → управляющие карты |
| 5 | Определение стиля | пресет-промпт / YandexGPT / IP-Adapter | стиль задаёт пользователь |
| 6 | Генерация мебели | **self-hosted SDXL-inpaint + multi-ControlNet** [+опц. LaMa-предочистка] | пишет мебель только в маске пола; ControlNet держит перспективу |
| 7 | Апскейл, QA, выдача | Real-ESRGAN + авто-QA | 2K–4K, проверка окон/масштаба/коллизий → Object Storage → вебхук → iOS |

### 3.3. Как встроен ArrangementEngine (почему нет «летающих диванов»)

`ArrangementEngine.arrange(items:room:)` выдаёт `placedItems` с `position` (метры), `rotation`, `size` +
`FloorBounds`. Механика: (1) движок считает план — *где* стоит каждый предмет, детерминированно, по
нормам `DesignNorms`; (2) 3D-боксы проецируются в перспективу фото по depth-карте → три conditioning-карты
(дополненная глубина, контуры предметов, per-item inpaint-маски); (3) SDXL-inpaint «одевает» мебель
**внутри заданной геометрии**, стиль отвечает только за внешний вид. Итог: **расстановку контролирует
движок (повторяемо, по нормам), диффузия лишь делает её фотореалистичной.** У конкурентов мебель ставит
сама нейросеть «на удачу» — отсюда кривой масштаб и предметы в стенах.

### 3.4. Вердикт по движку и стоимость

**Self-hosted SDXL 1.0 inpainting + ControlNet (depth+mlsd) на Yandex Compute GPU.** SDXL предпочтительнее
Flux: зрелее экосистема ControlNet (5+ против 3), влезает в 12–16 ГБ, ~25–40 сек/кадр; Flux.1 Fill —
апгрейд для премиум-листингов (24+ ГБ, веса некоммерческие — проверять). RU-альтернатива внутри того же
self-host — **Kandinsky 3.1** (Apache-2, отечественный). Себестоимость кадра: **7–11 ₽ на A100 / 0,5–2 ₽
на T4**. Экономика: батчинг, пул прерываемых ВМ, autoscale-to-zero, LaMa на CPU для предочистки.

### 3.5. MVP vs полный пайплайн

- **MVP** (доказать unit-экономику и quality-gate): SDXL-inpaint + 1–2 ControlNet (depth обяз.) +
  SAM2-маски + Depth Anything V2 + 5–6 фиксированных РФ-пресетов + ArrangementEngine + Real-ESRGAN. Одна
  GPU (T4 или прерываемый A100 / DataSphere). **Без** LaMa, IP-Adapter, LoRA, Flux, видео-облёта.
- **Полный** (после валидации спроса): + LaMa-предочистка занятых комнат + multi-ControlNet с
  откалиброванными strength + Style-LoRA + IP-Adapter (премиум) + итеративный inpaint + Flux + 4K-апскейл
  + пул прерываемых ВМ с батчингом.

### 3.6. Качество и Gate

- **Авто-QA** (на каждый кадр, дёшево): резкость (Laplacian), проверка что окна/стены не перекрашены
  (сравнение с protected-масками), масштаб мебели относительно `FloorBounds`, отсутствие коллизий
  (метрики из ThreedFront), реалистичность теней → при провале авто-повтор с другим seed.
- **Человеческий Gate (KPI):** на 50–100 реальных РФ-фото — 2–3 оценщика (лучше реальные
  риелторы/дизайнеры) бинарно «вставил бы в листинг: да/нет». **KPI ≥ 60% «да»** для прохождения MVP;
  замерять отдельно по ICP (риелтор vs дизайнер) и типам комнат.
