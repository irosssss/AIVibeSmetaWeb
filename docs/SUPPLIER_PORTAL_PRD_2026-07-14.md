## 1. Что изменилось с июньской стратегии

`PARTNER_STRATEGY.md` писалась, когда спрос = B2C-пользователь с AR-примеркой.
Сейчас спрос = **соло-дизайнер со сметой FF&E**. Это меняет почти всё на стороне ценности:

| Было (июнь, AR) | Стало (июль, смета дизайнера) |
|---|---|
| Ценность фабрике: AR-примерка у конечника | Ценность: **попадание в сметы дизайнеров** («Specified, not just seen» — формула Programa) |
| 3D-пайплайн GLB→USDZ — ядро онбординга | 3D **не нужен** для MVP; нужны фото silo+lifestyle, цены, габариты |
| CPA 12% с продажи | **Лиды и спецификации** (RFQ/PO уже есть в продукте); CPA — апсайд, не фундамент (урок Faradise не отменён) |
| Метрика фабрики: заказы из AR | Метрики: **раз в сметах / в шарах клиенту / в PO / RFQ-запросы** |

Что из июньского документа **остаётся в силе как есть**:
- §3 тиры фабрик и фокус на средних (Пенза/Владимир/МО/НН);
- §4.1 три формата фида: YML / CommerceML (1С) / CSV+фото;
- §5.1 модель атрибутов Saleor (типизированные атрибуты-фасеты) + Medusa (option→variant);
- §7.4 подписка фабрики должна быть скромной (Roomle €850+/мес — не про РФ);
- уроки Faradise: не комиссия как фундамент, не vanity-метрики, узкое место — спрос.

## 2. Ориентир: Programa Trade Portal for Brands

Проверено по их публичным страницам (14.07):

1. **Заявка → одобрение** («Submit your brand for approval») — листинг только после модерации. ~300 брендов, только ANZ.
2. **Каталог**: API или ручной ввод; поля — цвета, материалы, отделки, сроки поставки, описания, фото.
3. **Коллекции** — витрина топ-товаров + история бренда.
4. **Петля**: дизайнер добавляет товар в смету одним кликом; бренд получает спецификации и лиды.
5. **Аналитика бренду**: «Track leads, quotes, and orders directly tied to your products» — воронка по своим товарам.
6. **Не маркетплейс**: транзакций через платформу нет, дизайнер — главный клиент, бренд платит за присутствие/лиды.

Аналоги того же паттерна: Designer Pages PRO (265k товаров / 8 800 производителей),
Fohlio, Techstyles. Все — «каталог + лиды», не транзакции.

## 3. Инварианты (не обсуждаются)

1. **Мы не маркетплейс** (SOUL.md): деньги дизайнер↔поставщик ходят мимо нас. Поставщик получает спецификации/лиды, не «заказы через платформу».
2. **Деньги дизайнера — тайна и от поставщика тоже**: себестоимость, наценка, цена клиента поставщику не видны. Поставщик видит только факт спецификации своего товара и RFQ/PO, адресованные ему. Это расширение главного инварианта на третью аудиторию (в бенчмарке Programa уже помечено: видимость по аудиториям клиент/подрядчик/поставщик).
3. **Фото и данные грузит сам поставщик** — правило «чужие фото не рехостим» не нарушается: это его контент по оферте.
4. **Не строить витрин**: каждый этап включается только когда есть читатель данных (реальный дизайнер и реальный поставщик).

## 4. Роли и флоу онбординга

```
ПОСТАВЩИК                    ПЛАТФОРМА (владелец)             ДИЗАЙНЕР
  заявка (форма) ──────────►  модерация: одобрить/отклонить
  фид YML/1С/CSV ──────────►  валидация → каталог (YDB)
  карточка бренда, коллекции   scoping по supplier_id           поиск в библиотеке,
                                                                 «в смету одним кликом»,
  ◄── метрики: в сметах / в шарах / RFQ / PO ────────────────── drag&drop (есть с PR #28)
```

- **Заявка**: название, ИНН, контакты, категории, город/логистика. Модерация владельцем (как у Programa) — руками, без автоматики.
- **Кабинет поставщика (self-serve)** — этап V1+, не MVP (открытый вопрос §12.2 июньской стратегии решаем в пользу «сначала руками»: первых 5–10 поставщиков онбордит владелец по фиду, кабинет строим по живой боли).
- **Дизайнеру** портальный каталог доступен из существующего пикера библиотеки — как второй таб «Каталог поставщиков» рядом со «Моей библиотекой» (точка влияния, не остров).

## 5. Модель данных

Расширение существующей схемы, не параллельная:

- **Поставщик** = запись адресной книги (K5a) + новые поля: `verified` (прошёл модерацию), `inn`, `logo`, `about`, `leadTimeDefault`. Связь с позициями остаётся по имени (`supplierMatch`) для локальных, по `supplier_id` — для портальных.
- **Товар** = `blankProduct` (артикул/материал/габариты/срок/фото/rrp/unit — всё уже есть) + новое:
  - **`variants[]`** (Medusa-стиль): вариант = свой артикул + цвет (`color` + hex) + опционально своя цена/фото. Сейчас у нас товар одномерный — «свой артикул, цвета» поставщика требуют вариантности. В смету попадает конкретный вариант (плоско, как сейчас), вариантность живёт в каталоге.
  - `brand` (бренд ≠ поставщик: дилер может продавать чужой бренд);
  - `category/subcategory/room` как фасеты (Saleor-стиль, §5.2 июньской);
  - `inventory`/`priceDate` — свежесть (паспорт свежести цены уже есть в библиотеке, B3).
- **Цены в каталоге**: поставщик публикует **RRP (розницу)** и опционально **трейд-цену для дизайнеров** (закрыта от анонимов). RRP-слой в смете уже есть (п.17); трейд-цена поставщика ложится в себестоимость позиции при добавлении.

## 6. Метрики поставщику (кабинет V1)

Формула Programa «Specified, not just seen» — показываем воронку, не показы:

1. Сколько раз товар добавлен в сметы (без имён проектов и без денег дизайнера).
2. Сколько раз ушёл в шару клиенту (= товар дошёл до конечника).
3. RFQ-запросы и PO на поставщика (это и есть лиды — механика PO уже в продукте, K5d; RFQ ждёт домена).
4. Карточка: полнота данных, битые фото, протухшие цены.

## 7. Этапность

| Этап | Что | Условие входа |
|---|---|---|
| **V0 — руками** | Владелец онбордит 5–10 средних фабрик по фиду (YML/1С/CSV → скрипт-импорт в YDB); дизайнер видит «Каталог поставщиков» в пикере | Yandex Cloud + первые живые дизайнеры |
| **V1 — кабинет** | Self-serve: заявка, загрузка фида, карточка бренда, метрики §6 | ≥5 активных поставщиков V0 и подтверждённый спрос дизайнеров |
| **V2 — авто-фид** | API/регулярная синхронизация фида, авто-валидация, коллекции | Живая боль V1 «фид протухает» |

Монетизация поставщика (скромная подписка + лиды; CPA — только апсайд) — цифры не фиксируем до V0: сначала ценность, потом прайс (открытый вопрос §12.3 июньской остаётся владельцу).

## 8. Не делаем

- Транзакции/оплату товара через нас; корзину; «заказы» как сущность платформы.
- 3D/AR-пайплайн — вне рамки веб-сметы (июньские §4.3/§9.1 замораживаются, не удаляются).
- Публичный B2C-каталог/SEO-ферму карточек товаров (мы не INMYROOM).
- Рейтинги/отзывы поставщиков на старте — нечем наполнить честно.

## 9. Открытые вопросы владельцу

1. Порог модерации V0: берём всех подряд из фокус-тиров или только «средние с фото silo»?
2. Трейд-цена в каталоге: обязательное поле или опция? (Без неё ценность каталога для дизайнера — только удобство, с ней — деньги.)
3. Показывать ли поставщику город/студию дизайнера в лиде (RFQ/PO) до ответа — или только после взаимного согласия?
